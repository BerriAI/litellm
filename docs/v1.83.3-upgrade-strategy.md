# LiteLLM v1.83.3-stable Upgrade Strategy

**Author:** Shriharsha M
**Date:** 2026-04-21
**Target:** `juspay/litellm` `main` → upstream `v1.83.3-stable`
**Current base:** `juspay/litellm` is at upstream `v1.81.3-stable` + 87 custom commits.

---

## TL;DR

Reset `feature/upgrade-to-litellm-v1.83.3` to upstream `v1.83.3-stable`, then cherry-pick 87 custom commits back onto it in 9 domain-grouped batches, auditing each commit against upstream changes before replay. Squash-merge the final PR to `main` to match the pattern of prior upgrades (PR #52, PR #68). Strict drop-gate: no custom patch is dropped unless upstream demonstrably covers the same behavior with equivalent semantics and a verification test exists.

## Section 2 — Phased process

### Phase 0 — Prep & safety nets (~30 min)

```bash
git fetch upstream --tags
git fetch origin

# Dated backup of main
git branch backup/pre-v1.83.3-upgrade-$(date +%Y%m%d) main
git push origin backup/pre-v1.83.3-upgrade-$(date +%Y%m%d)

# Use the existing branch, stack a scratch audit branch on top
git checkout feature/upgrade-to-litellm-v1.83.3
git checkout -b audit/v1.83.3

# Create artifact directories
mkdir -p .upgrade/audit .upgrade/batches scripts/upgrade
```

**Exit gate:** backup branch pushed to origin, working tree clean, audit branch created.

### Phase 1 — Inventory & domain grouping (~1 day)

Generate the replay matrix and cluster into 9 domain batches.

```bash
git log --reverse --no-merges --format="%H|%ad|%s" --date=short \
  88c78df665^..main ^ed2b8e9fae ^7fc45ccc1e \
  > .upgrade/replay-matrix.csv
```

| # | Batch | Approx # commits |
|---|---|---|
| 1 | Claude / Anthropic compatibility (header drops, content block, reasoning field) | 8–10 |
| 2 | GCS + GCP logging (loggers, directory, disable header, BigQuery) | 10–12 |
| 3 | Routing strategies (least-busy, sticky, v2, simple-shuffle, vision fallback) | 10 |
| 4 | Rate-limit & concurrency (MPR, counter leak, drift, perf) | 8 |
| 5 | Budgets (`FREE_MODELS`, user budget restriction, reset job, duration) | 6 |
| 6 | Analytics endpoints (DAU/WAU/MAU, leaderboard, spend, failure logs) | 10 |
| 7 | UI features (filters, dropdown, message filter, pagination, permissions) | 10 |
| 8 | Admin / user mgmt (user delete allowlist, team update, audit logging, bulk cost) | 6 |
| 9 | Build / CI / Playground / misc (GCR, Jenkins, Dockerfiles, Prisma, playground) | 10 |

Save per-batch commit lists to `.upgrade/batches/NN-<name>.txt`.

**Exit gate:** Every one of the 87 commits classified into exactly one batch.

### Phase 2 — Per-batch audit (~3–5 days)

For each batch, produce `.upgrade/audit/batch-NN-<name>.md` following the record format in Section 3. Start with highest-risk domains: rate-limit, routing, budgets.

**Exit gate:** All 9 batch audits written and reviewed. Zero `DROP` rows without a justification passing the four-clause drop-gate.

### Phase 3 — Replay (~2–4 days)

```bash
# Reset working branch to upstream target
git checkout feature/upgrade-to-litellm-v1.83.3
git reset --hard v1.83.3-stable

# Cherry-pick one batch at a time
for sha in $(cat .upgrade/batches/01-claude-compat.txt); do
  git cherry-pick -x "$sha" || { echo "conflict on $sha"; break; }
done

# Smoke-test, then checkpoint
make lint-ruff
make test-unit
git tag replay-checkpoint/01-claude-compat
```

Rules:
- One batch at a time. Do not start batch N+1 until batch N's smoke tests pass.
- Skipped commits (`DROP` decisions) are recorded via `git notes add` with an audit-doc link.
- If a commit classified as `KEEP-AS-IS` hits a conflict, **stop** — that's an audit miss; re-audit before resolving.
- Use `git cherry-pick -x` so the target commit message carries the original SHA.

**Exit gate:** all 9 batches applied, 9 checkpoint tags exist, `make lint` and `make test-unit` green.

### Phase 4 — Verification (~2–3 days)

Run the MUST-SURVIVE checklist end-to-end on a sandbox deploy. See Section 4 for concrete tests per item.

**Exit gate:** every MUST-SURVIVE item has a green check with evidence.

### Phase 5 — Ship (~1 day)

- Open PR `feature/upgrade-to-litellm-v1.83.3 → main`.
- PR description: link audit docs, list dropped commits with justifications, attach verification evidence.
- **Squash-merge** (matches #52 / #68 convention).
- Tag the merge: `git tag -a v1.83.3-juspay-sandbox` (mirrors sandbox pattern from #68).
- Deploy to sandbox, run burn-in period, then promote.
- Retain `backup/pre-v1.83.3-upgrade-*` branches until burn-in passes.

**Wall-clock estimate:** 7–13 working days, dominated by Phase 2 audit and Phase 3 replay.

## Section 3 — Audit methodology (keep / rework / drop gate)

### Per-commit decision tree

```
For each of the 87 commits:

  1. Compute "what did this commit actually change?"
       git show <sha> --stat
       git show <sha>                 # full diff
     → files touched, functions touched, intent (from message + diff)

  2. Compute "what did upstream do in the same region since v1.81.3?"
       git log --oneline v1.81.3-stable..v1.83.3-stable -- <files>
       git diff v1.81.3-stable v1.83.3-stable -- <files>

  3. Apply the decision tree:

     Upstream did NOT touch the same region
       → KEEP-AS-IS
         Evidence: empty/irrelevant upstream log for those files.
         Expected replay: clean cherry-pick.

     Upstream touched the region, but solves a DIFFERENT problem
       → REWORK (keep our intent, adapt to upstream context)
         Evidence: side-by-side function-level diff.
         Expected replay: conflict; merge both intents.

     Upstream touched the region AND addresses our commit's intent
       → DROP candidate — must pass the drop-gate below.

     Upstream REMOVED / RENAMED the surface we patched
       → REWORK — find the new surface; port the patch.
         Never DROP just because the surface moved.
```

### Drop-gate (strict — all four must be true)

1. **Behavior equivalence proven** — specific upstream commit/PR cited with URL or SHA.
2. **No semantic regression** — upstream solution doesn't change observable behavior (env vars, Redis schema, headers, metric names).
3. **Verification test exists** in the Phase 4 MUST-SURVIVE list.
4. **Human sign-off recorded** — reviewer named on the audit line.

If any one is false → the default is `KEEP-AS-IS` or `REWORK`. "Probably fine" is not a valid drop reason.

### Record format (one row per commit)

```markdown
## <short-sha> — <commit subject>
- files: <...>
- intent: <...>
- upstream in region since v1.81.3:
  - <url_1> <summary>
  - <url_2> <summary>
- decision: KEEP-AS-IS | REWORK | DROP
- rationale: <...>
- replay plan: <clean cherry-pick | conflict resolution notes | dropped commit reference>
- verification: <MUST-SURVIVE line item(s)>
- reviewer: @<handle>
```

### Domain-specific audit hazards

Start the audit with these high-churn domains — unblocking them early avoids late surprises.

| Domain | Custom surface | Upstream churn pattern |
|---|---|---|
| Rate-limit / MPR | `parallel_request_limiter.py`, `dual_cache.py` | Upstream rewrote to v2 limiter; multiple leak fixes. **High REWORK probability.** |
| Routing | `router_strategy/least_busy.py`, `lowest_tpm_rpm_v2.py` | Upstream added sticky TPM routing, moved cooldown logic. Sticky-least-busy likely REWORK. |
| Budgets | `user_api_key_auth.py`, `auth_checks.py`, `reset_budget_job.py` | Upstream added org/team budget checks and reset-job refactor. `FREE_MODELS` bypass needs careful REWORK. |
| GCS / GCP logging | `gcs_bucket/gcs_logger.py`, custom stdout logger | Upstream extended GCS logger — directory overlap → REWORK. Custom stdout logger is our own file → likely KEEP-AS-IS. |
| Spend / analytics | `db_spend_update_writer.py`, `user_agent_analytics_endpoints.py` | Upstream added spend-log aggregation; some custom failure-log aggregation may be redundant → DROP candidate, evaluate carefully. |
| UI / Next.js | `litellm/proxy/_experimental/out/**` | Heavy upstream UI churn; expect per-component REWORK. `KEEP-AS-IS` unlikely. |

### Tooling shortcuts

- `scripts/upgrade/audit-inspect <sha>` — runs `git show <sha> --stat`, `git log --oneline v1.81.3-stable..v1.83.3-stable -- <files>`, opens side-by-side diff.
- `scripts/upgrade/batch-replay <batch-file>` — wraps `git cherry-pick -x` per commit, stops on conflict, appends notes to audit doc.

Both are disposable; delete after the upgrade.

### Backup branches (mandatory before any destructive op)

Before Phase 3 `git reset --hard`:

```bash
git branch backup/pre-v1.83.3-upgrade-$(date +%Y%m%d) main
git branch backup/feature-branch-pre-reset-$(date +%Y%m%d) feature/upgrade-to-litellm-v1.83.3
git push origin backup/pre-v1.83.3-upgrade-$(date +%Y%m%d) \
                 backup/feature-branch-pre-reset-$(date +%Y%m%d)
```

Retain both backups until the upgrade passes prod burn-in. Do not delete.

### Rollback criteria (abort conditions)

Abort the replay and revert to backup if any of:

- Phase 3 cherry-pick conflict rate > 50% of commits (indicates audit was too optimistic; re-plan).
- Phase 4: any MUST-SURVIVE item cannot be made green within 1 day of debugging.
- Post-merge sandbox burn-in: elevated error rate (>2× baseline) or P99 latency regression >30% sustained >15 min.

Rollback mechanics:
```bash
# Local revert before merge
git reset --hard backup/feature-branch-pre-reset-YYYYMMDD

# Post-merge revert (if already squash-merged)
git revert <squash-merge-sha> -m 1
```

### Burn-in plan

| Stage | Duration | Acceptance |
|---|---|---|
| Local + unit tests | immediate | `make lint` + `make test-unit` green per batch |
| Sandbox deploy | 48–72h | MUST-SURVIVE checklist green; no elevated 5xx |
| Canary (10% prod) | 24–48h | P95 latency, error rate, Redis counter health match baseline |
| Full prod promote | — | only after canary clean |

### Post-upgrade cleanup

- Delete `scripts/upgrade/` helper scripts.
- Delete `.upgrade/` artifacts or move to `docs/archive/v1.83.3-upgrade/`.
- Keep `backup/pre-v1.83.3-upgrade-*` branch for at least 30 days after prod promote.
- Open a next-milestone checklist: are there any customizations still needed that upstream didn't adopt? File issues upstream for the durable ones.
